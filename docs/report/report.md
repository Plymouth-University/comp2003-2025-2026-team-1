# Introduction

![Cover art for the game.](images/co-operation%20title.png)

Co-Operation: Multiturn is a game created by Mind Feast Games (founded in 2020) that is centred around a turn-based gameplay loop of healing and relocating patients within a comedic hospital setting and getting them the correct medicine in a timely manner. Mind Feast Games create games as part of an initiative to make games that encourage people to collaborate and connect with one another; something I feel they achieved with this initial goal. great aspect of the game is the integration of mobile devices to control the actual characters. It is implemented in a similar vein to the Jackbox games which allow for a queued set of actions which play out on the screen. Being able to join a created game lobby from phones, laptops, tablets or any device with a browser makes the gameplay loop portable in how you can take the essentially anywhere after having booted the game. The movement has to be co-ordinated in an active way for the game to work smoothly. The discussion between a group is very important and can result in some comical moments when a wrong move is chosen!

![Initial preview of game.](images/level%20preview%20intro.png)

We played the game as a group so we could gauge the gameplay loop to see how the levels played out. We found a lot of heart in the game and the level design complimented the game loop across the base game’s levels. We had a lot of fun dissecting the various avenues available to the player and looking at the way each of the props allowed the characters traverse the levels differently. For example, there are low barriers which allow for items and patients to be thrown over but there’s also ones which completely stop the player from going through them. We began thinking up some potential methods for how we’d approach the generation at this stage. 

![Initial preview of game.](images/drawn%20players%20playing%20game.png)

The brief laid out by our client, Mind Feast Games, states that we’d need to create AI tools in order to create an arbitrary amount of levels which should ideally be cohesive and fun to play, and that fit the general quality of the base game levels. It was intended to be an external tool to generate each levels for use in the base game. Since the levels would be generated externally, they’d be hand picked by the developers to be included in the final game to be shipped which would add to the release. We’d need to take extra care in making sure the levels were fun and accessible to players who are familiar/fans of the base game’s levels; making sure the generated levels were playable alongside them was important to us as a team.

# Conclusion

Our team managed to create a gui-based level editor, 6 levels, python scripts for automation.

The original requirements were to build an AI tool to generate levels and we were unable to do that. However, we have made tools to make the level design quicker and provided python scripts that have a skills.md which can be used to train AI models in the future for full automation.
